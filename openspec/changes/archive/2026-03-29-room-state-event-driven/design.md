## Context

当前系统中，租约和房间状态之间存在业务关联，但没有实现自动化的状态同步。租约状态变化（激活、退租、过期）需要手动更新房间状态，这容易导致数据不一致。

系统已经具备基本的领域事件基础设施：
- 聚合根可以记录和发布领域事件
- 事件总线可以分发事件
- 已有操作日志事件处理器作为示例

### 约束条件
- 需要保持与现有代码的兼容性
- 不能修改已有的DDD架构重构成果
- 需要遵循现有的代码风格和模式

## Goals / Non-Goals

**Goals:**
- 当租约激活时，自动将对应房间状态设置为已出租
- 当租约退租或过期时，自动将对应房间状态设置为可出租
- 所有状态变化都通过领域事件记录
- 保持系统的最终一致性

**Non-Goals:**
- 不修改租约或房间的核心业务逻辑
- 不修改现有的命令处理器架构
- 不引入新的外部依赖

## Decisions

### 1. 使用事件驱动架构实现状态同步
**决定**: 使用领域事件来实现租约状态与房间状态的同步

**理由**:
- 系统已经有完整的领域事件基础设施
- 符合DDD原则，降低聚合根之间的直接耦合
- 保证状态变化的可追踪性
- 便于后续扩展（如添加审计日志、通知等）

**备选方案**:
- 直接在租约命令处理器中调用房间仓储 - 会造成耦合，不符合DDD原则

### 2. 在房间模型中添加状态变更事件
**决定**: 修改房间模型的MarkRented和MarkAvailable方法，使其记录领域事件

**理由**:
- 符合聚合根产生事件的DDD原则
- 保证状态变化与事件记录的原子性
- 与现有房间创建、更新事件的模式保持一致

### 3. 创建专门的事件处理器
**决定**: 创建一个新的事件处理器来监听租约事件并更新房间状态

**理由**:
- 单一职责原则，每个处理器负责一种功能
- 便于测试和维护
- 与现有的操作日志处理器模式保持一致

## Risks / Trade-offs

### [风险] 事件处理失败导致状态不一致
**影响**: 租约状态已更新，但房间状态未同步
**缓解措施**:
- 事件处理器中添加重试逻辑
- 记录详细的错误日志
- 考虑添加定期的一致性检查任务

### [风险] 并发操作导致竞态条件
**影响**: 多个租约操作同时影响同一个房间
**缓解措施**:
- 利用数据库事务保证原子性
- 考虑添加乐观锁机制

## Migration Plan

1. 修改房间模型，添加状态变更事件
2. 创建新的事件处理器
3. 更新依赖注入配置
4. 测试状态同步功能
5. 部署到生产环境

**回滚策略**: 可以直接回滚代码变更，系统会退回到手动管理状态的方式。
