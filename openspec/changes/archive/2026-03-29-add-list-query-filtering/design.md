## Context

项目是一个基于 DDD（领域驱动设计）+ CQRS（命令查询职责分离）+ 事件驱动架构的租赁管理系统。当前所有列表页面（房东、租约、账单、位置、房间、打印）都直接展示全部数据，没有查询筛选和分页功能，在处理大量数据时用户体验不佳。

操作日志查询功能已经实现了完整的查询和分页功能，可以作为参考实现。

## Goals / Non-Goals

**Goals:**
- 为所有列表页面添加查询功能，支持根据查询条件筛选数据
- 实现后端分页功能，提高大数据量场景下的性能
- 统一查询接口和数据格式，保持代码一致性
- 提供良好的用户交互体验，包括查询表单、结果展示和分页控件

**Non-Goals:**
- 不实现前端分页（全部数据加载到前端后分页）
- 不实现复杂的联合查询（多表关联查询）
- 不实现高级查询功能（如模糊匹配、正则表达式查询）

## Decisions

### 1. 查询架构模式
**决定：** 采用与操作日志查询相同的 CQRS 查询模式

**理由：**
- 保持与现有代码库架构一致
- 分离查询模型与领域模型，提高查询性能
- 已经有成熟的参考实现

### 2. 查询参数格式
**决定：** 使用统一的查询参数格式，包括查询条件和分页参数

**参数结构：**
```go
type ListQuery struct {
    // 查询条件（根据不同实体类型定义）
    // ...

    // 分页参数
    Offset int    `json:"offset"`
    Limit  int    `json:"limit"`
}

type QueryResult struct {
    Items []interface{} `json:"items"`
    Total int           `json:"total"`
    Offset int          `json:"offset"`
    Limit  int          `json:"limit"`
}
```

**理由：**
- 统一的参数格式简化了前端开发
- 明确分离查询条件和分页参数，提高代码可读性
- 支持扩展更多查询条件

### 3. 查询实现技术
**决定：** 使用 SQLite 的 LIKE 和 BETWEEN 操作符实现查询功能

**查询方法：**
- 模糊搜索：使用 `LIKE '%value%'`
- 时间范围查询：使用 `BETWEEN`
- 分页：使用 `OFFSET` 和 `LIMIT`

**理由：**
- SQLite 支持这些操作符，性能满足当前需求
- 实现简单，开发成本低
- 与现有代码库的查询风格一致

### 4. 前端查询组件
**决定：** 使用 Ant Design 的查询组件和表格组件

**组件使用：**
- 查询表单：使用 `Form` 组件
- 查询按钮：使用 `Button` 组件
- 表格：使用 `Table` 组件
- 分页：使用 `Pagination` 组件

**理由：**
- 项目已经使用 Ant Design，保持一致性
- Ant Design 提供了丰富的查询和表格组件
- 组件功能完善，用户体验良好

## Risks / Trade-offs

### 1. 查询性能风险
**风险：** 在处理大量数据时，查询可能会变慢

**缓解策略：**
- 实现后端分页，只返回需要的数据
- 为常用查询字段添加索引
- 优化 SQL 查询语句

### 2. 查询条件复杂度风险
**风险：** 不同实体类型的查询条件差异较大，可能导致代码重复

**缓解策略：**
- 抽象通用查询逻辑
- 使用查询构建器模式
- 保持查询模型简单，只包含必要的查询条件

### 3. 前端代码复杂度风险
**风险：** 每个列表页面都需要实现查询功能，可能导致代码重复

**缓解策略：**
- 创建可复用的查询组件
- 统一查询表单和表格的布局
- 使用 Hook 封装查询逻辑
