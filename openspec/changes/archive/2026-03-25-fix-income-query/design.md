## Context

收入查询页面显示各个收入类型的明细和占比。目前存在以下问题：
1. 当总金额为 0 时，各个收入类型的占比计算会出现 NaN（0 作为除数）
2. 押金支出作为负数，但在计算占比时没有正确处理
3. 数据类型转换过程中存在精度丢失问题

## Goals / Non-Goals

**Goals:**
- 修复占比计算出现 NaN 的问题
- 优化押金支出的显示和计算逻辑
- 确保所有数值计算的安全性和准确性
- 改进用户界面的数据展示

**Non-Goals:**
- 不修改后端数据结构
- 不改变收入类型的定义
- 不影响其他功能模块

## Decisions

### 1. NaN 问题修复
- **问题分析**: 当 total 为 0 时，计算 percentage 会出现 0/0 导致的 NaN
- **解决方案**: 在计算占比前检查 total 是否大于 0，否则返回 '0%'

### 2. 押金支出计算优化
- **问题分析**: 押金支出是负数，但当前代码直接将其作为正数计算占比
- **解决方案**: 在计算占比和显示时，需要正确处理负数金额

### 3. 数据格式化改进
- **问题分析**: 金额格式化过程中可能存在精度问题
- **解决方案**: 使用更安全的格式化方法，确保数值精度

## Risks / Trade-offs

**风险 1**: 修复后的占比计算可能与修复前有细微差异
- **缓解**: 使用标准的数学库函数，并确保所有边界条件都被覆盖

**风险 2**: 押金支出的负数值可能影响表格数据排序
- **缓解**: 确保在排序和计算时正确处理负数逻辑
