## Context

当前项目是一个基于 DDD 架构的房屋租赁管理系统，已有完整的事件总线机制和多个领域模型（房间、租赁、房东等）。系统缺乏统一的操作日志审计能力，无法追踪领域事件的完整历史。

**现有架构组件：**
- 事件总线（Event Bus）：支持事件发布和订阅
- SQLite 持久化层：已有多个领域表
- React + TypeScript 前端：已有多个管理页面
- 查询总线：支持查询操作

## Goals / Non-Goals

**Goals:**
- 为所有领域事件提供完整的操作日志记录
- 支持按领域类型、时间范围、聚合ID查询操作日志
- 提供前端查询界面，展示完整日志详情
- 不影响现有领域模型和事件处理流程
- 日志记录性能开销可接受

**Non-Goals:**
- 不修改现有领域模型
- 不实现复杂的日志分析功能
- 不实现日志告警或通知机制
- 不支持实时日志流（可作为后续改进）

## Decisions

### 决策1：使用事件监听器模式记录日志

**选择：** 实现专门的 `OperationLogHandler` 监听所有事件

**原因：**
- 与现有事件总线完全契合
- 不修改现有领域模型代码
- 易于扩展和测试
- 符合关注点分离原则

**替代方案：**
- 事件总线中间件：会影响所有事件处理，耦合度高
- 领域服务中直接记录：需要修改现有代码，违反开闭原则

### 决策2：SQLite 存储操作日志

**选择：** 使用现有 SQLite 数据库，新增 `operation_logs` 表

**原因：**
- 无需引入新的技术栈
- 与现有持久化架构一致
- 查询性能足够
- 事务支持完善

**替代方案：**
- 文件日志：查询能力差，不适合结构化查询
- 事件存储：需要引入新组件，复杂度高

### 决策3：异步记录日志

**选择：** 使用 `PublishAsync` 方式处理日志记录

**原因：**
- 不阻塞主业务流程
- 降低对性能的影响
- 即使日志记录失败也不影响主要操作

**替代方案：**
- 同步记录：可能影响响应时间

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 日志记录性能开销 | 中等 | 使用异步处理，定期监控性能 |
| 日志数据快速增长 | 中等 | 考虑后续添加日志清理或归档策略 |
| 事件内容敏感信息泄露 | 高 | 确保不记录密码等敏感信息，可添加字段过滤 |
| 查询性能下降 | 低 | 为常用查询字段添加索引 |

## Data Model

**OperationLog 表结构：**

| 字段 | 类型 | 说明 | 索引 |
|------|------|------|------|
| id | TEXT | 主键 | PK |
| timestamp | DATETIME | 操作发生时间 | INDEX |
| event_name | TEXT | 事件名称 | INDEX |
| domain_type | TEXT | 领域类型（room/lease/landlord等） | INDEX |
| aggregate_id | TEXT | 关联的聚合根ID | INDEX |
| operator_id | TEXT | 操作人ID | INDEX |
| action | TEXT | 操作类型（created/updated/deleted） | |
| details | JSON | 详细数据（事件完整内容） | |
| metadata | JSON | 元数据（请求来源、IP等） | |
| created_at | DATETIME | 创建时间 | INDEX |

## Migration Plan

1. 创建数据库迁移脚本添加 `operation_logs` 表
2. 实现 OperationLog 领域模型和存储仓库
3. 实现 OperationLogHandler 事件处理器
4. 在事件总线中注册处理器
5. 实现查询API
6. 实现前端查询页面
7. 测试并部署

**回滚策略：**
- 可直接移除 OperationLogHandler 注册，不影响其他功能
- 数据库表可保留，不影响现有查询
