## Why

当前应用层的逻辑使用了传统的服务模式，存在以下问题：
1. 应用服务直接耦合到领域逻辑，缺乏明确的职责边界
2. 命令和查询没有明确分离，不符合 CQRS 原则
3. 没有使用事件驱动架构，系统的异步处理和扩展性受限
4. 存在一些未使用或冗余的服务，增加了维护复杂度

通过引入完整的总线架构（commandbus、querybus、eventbus），可以实现：
1. 更清晰的架构边界
2. 更好的关注点分离
3. 更强大的异步处理能力
4. 简化的代码维护
5. 更好的系统可扩展性

## What Changes

- **重构**：将所有应用层逻辑替换为 commandbus、querybus 和 eventbus 管理
- **移除**：删除未使用或冗余的服务，如过时的应用服务实现
- **优化**：重构 HTTP 处理器，使其成为简单的命令/查询分发器
- **增强**：引入事件驱动架构，实现命令执行后的领域事件发布
- **改进**：优化查询逻辑，使其通过 querybus 统一处理

## Capabilities

### New Capabilities
- `query-bus`: 查询总线实现，用于统一处理所有查询操作
- `event-bus`: 事件总线实现，用于发布和订阅领域事件
- `bus-integration`: 总线架构在应用层的完整集成

### Modified Capabilities
- `command-bus`: 增强命令总线实现，支持更复杂的命令分发场景
- `ddd-architecture`: 更新架构规范，明确总线在系统中的位置

## Impact

- **后端代码**：
  - 重构 `internal/application/` 目录，引入 querybus 和 eventbus
  - 删除未使用的传统服务实现
  - 优化 HTTP 处理器，简化路由和请求处理
  - 在 `internal/application/cqrs/` 中引入事件发布机制
- **架构**：
  - 实现完整的 CQRS+ES（Event Sourcing）架构模式
  - 统一命令、查询和事件的处理方式
  - 增强系统的可扩展性和可维护性
- **代码量**：移除冗余服务会减少代码行数，但引入总线架构会新增一些基础设施代码
- **依赖**：可能需要新增一些异步处理和事件发布相关的依赖
- **测试**：需要重构现有的测试，特别是集成测试，以适应新的架构
