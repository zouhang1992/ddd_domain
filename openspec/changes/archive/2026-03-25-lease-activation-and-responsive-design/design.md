## Context

当前房屋管理系统是一个 DDD 架构的应用，前端使用 React + TypeScript + Ant Design，后端使用 Go + FX 依赖注入框架 + SQLite。

**当前状态：**
- 前端页面在不同屏幕尺寸下布局不够理想
- 租约列表缺少押金信息展示
- 缺少直接从租约列表打印合同的功能
- 租约无法从待生效状态转换为生效状态
- 收入汇总中缺少押金的收入（收取时）和支出（退租时）信息

**约束条件：**
- 保持与现有架构的一致性
- 使用 Ant Design 已有的响应式布局组件
- 不引入新的外部依赖

## Goals / Non-Goals

**Goals:**
- 实现前端页面的响应式设计，支持不同浏览器宽度的自适应布局
- 在租约列表中添加押金信息展示
- 实现租约列表的直接打印功能
- 实现租约生效的业务能力，支持将待生效租约转换为生效状态
- 在收入汇总中添加押金的收入（收取时）和支出（退租时）信息显示

**Non-Goals:**
- 不重构整体架构
- 不改变后端 API 的整体设计模式
- 不添加移动端的专门优化（只做基础响应式）

## Decisions

### 1. 响应式设计实现

**决定：** 使用 Ant Design 的 Grid 组件和响应式断点

**理由：**
- Ant Design 内置了完整的响应式系统
- Grid 组件支持 xs/sm/md/lg/xl/xxl 断点
- 与现有项目技术栈一致

**实现方案：**
- 使用 Row/Col 组件的响应式配置
- 设置表格的 scroll.x 属性，在小屏幕下可横向滚动
- 使用 Card 的响应式样式
- 对于表单，在小屏幕下调整布局为单列

### 2. 租约列表押金信息展示

**决定：** 在租约列表表格中新增押金列

**理由：**
- 押金是租约的重要信息，用户需要快速查看
- 表格布局直观，易于理解
- 与其他信息展示方式一致

**实现方案：**
- 新增押金状态和金额列
- 使用 Tag 组件显示押金状态
- 添加押金金额格式化显示

### 3. 租约打印功能实现

**决定：** 使用现有的 `/leases/{id}/contract` 接口直接下载

**理由：**
- 后端已经有下载接口
- 直接下载比 iframe 预览更简单
- 保持与账单打印的一致性

**实现方案：**
- 在租约列表操作列添加"打印合同"按钮
- 点击后直接调用下载接口
- 添加加载状态提示

### 4. 租约生效功能

**决定：** 新增激活租约的 API 端点和业务逻辑

**理由：**
- 需要明确的用户触发动作来生效租约
- 保持与现有命令总线架构的一致性
- 需要验证业务规则

**实现方案：**
- 新增 `activate_lease` 命令
- 新增 `PUT /leases/{id}/activate` API 端点
- 验证租约状态只能从 "pending" 转换为 "active"
- 触发相关领域事件

### 5. 押金收入/支出追踪

**决定：** 在收入汇总中添加押金的收取和退还追踪

**理由：**
- 押金是重要的资金流动，需要在收入汇总中反映
- 收取押金时应记录为收入
- 退还押金时应记录为支出（负收入）
- 用户需要完整的财务视图

**实现方案：**
- 在收入查询逻辑中包含押金类型的账单
- 押金收取（bill_type = 'deposit'）显示为正收入
- 押金退还（bill_type = 'checkout' 且包含 deposit_refund）显示为负收入
- 在收入页面新增押金收入/支出的分类显示
- 更新收入统计，区分租金、水电、押金等不同类型

## Risks / Trade-offs

**风险 1：响应式设计在极窄屏幕下仍可能有问题**
- **风险：** 超小屏幕（< 320px）下表格可能仍显示不佳
- **缓解：** 添加横向滚动作为兜底，提示用户使用更大屏幕

**风险 2：租约生效业务规则复杂**
- **风险：** 可能存在未考虑到的业务规则
- **缓解：** 参考参考项目的实现，充分测试边界情况

**风险 3：已有数据兼容性**
- **风险：** 现有租约数据可能与新状态不兼容
- **缓解：** 确保状态转换逻辑向前兼容，添加数据迁移支持
